# Liteboard — Design Spec

## Overview

Liteboard is an open-source tool that manages AI-driven development from brainstorm to built feature branch. It ships as two pieces:

1. **npm package** (`npm i -g liteboard`) — a modular orchestrator that spawns parallel agents in git worktrees, manages dependencies, merges results, and renders a live terminal dashboard.
2. **Claude Code plugin** — skill `.md` files that register `/liteboard:brainstorm`, `/liteboard:task-manifest`, and `/liteboard:run` as slash commands.

The workflow is three steps:
1. `/liteboard:brainstorm` — interactive design session, produces `design.md`
2. `/liteboard:task-manifest` — generates a task manifest from the design, produces `manifest.md`
3. `/liteboard:run <project>` — launches the orchestrator in Bash, enters supervisor mode

The orchestrator runs in the terminal with a live dashboard. The supervisor is baked into the `/liteboard:run` skill — Claude monitors progress, retries failures, fixes merge issues, and reports status. It's designed to run for hours in a single context window by staying lightweight.

**License:** MIT

## Core Principles

1. **Human owns the design.** The design doc is the source of truth. Time spent refining it saves time in the build phase.
2. **Agents own the build.** Once the manifest is generated and confirmed, the human watches — they don't drive.
3. **Shared memory bridges tasks.** No agent starts blind. Every task gets a live snapshot of what's been built so far.
4. **Explore before planning.** Every agent investigates existing codebase patterns before writing a single line.
5. **Never touch main.** Liteboard never commits to or pushes to `main`. All work happens on the feature branch. The user merges via PR.
6. **No blind agreement.** Every review loop is a real debate. Agents critically evaluate feedback — no performative agreement.
7. **Supervisor stays cheap.** The supervisor monitors for hours by reading small files and tailing logs — never deep-diving unless something fails.

## Skills (User-Invocable)

### `/liteboard:brainstorm`

Adapted from `superpowers:brainstorming`. Starts a new liteboard project.

- Explores the codebase for context
- Asks clarifying questions one at a time (multiple choice preferred)
- Proposes 2-3 approaches with trade-offs
- Writes `design.md` to `docs/liteboard/<auto-slug>/`
- Creates the project folder structure on first write
- Ends by prompting the user to run `/liteboard:task-manifest`

### `/liteboard:task-manifest`

Adapted from `task-manifest`. Takes a design doc and generates a build manifest.

- Reads `design.md` from the project folder
- Generates tasks with: id, title, creates, modifies, dependsOn, requirements, tddPhase, commitMessage, complexity
- Runs an architect review loop (critic evaluates completeness, security, DRY, dependency correctness — max 3 rounds)
- Writes `manifest.md` to the project folder
- Shows the user a summary (task count, dependency layers, high-complexity callouts)
- Ends by prompting the user to run `/liteboard:run`

### `/liteboard:run`

The orchestrator + supervisor skill.

- Invokes `liteboard run <project>` via the Bash tool
- The script runs in the terminal with a live dashboard (progress bar, task status, turn counts, last-line output)
- After launching, Claude enters supervisor mode (see Supervisor Design section)

## Brief Components (Agent Prompt Injection)

These are `.md` files in the `commands/` directory. Not user-invocable — they're composed into the prompt sent to each `claude -p` agent by `brief.ts` at spawn time. Spawned agents don't have access to the liteboard plugin, so everything they need is in their brief.

### `agent-orientation.md`

Tells the spawned agent:
- What liteboard is, what project it's building
- Its mandatory workflow: Explore -> Plan -> Plan Review -> Implement -> Code Review -> Commit
- How to use the Agent tool for explore subagents and review subagents
- Rules: don't touch unrelated files, commit with exact message, write memory entry
- Plan execution discipline (folded from superpowers:executing-plans):
  - Follow your plan step by step — don't skip ahead or reorder
  - Run every verification command specified — don't assume it passes
  - Stop immediately when blocked (missing dependency, unclear requirement, repeated test failure) — describe the blocker clearly and commit what you have rather than guessing
  - Mark each phase complete before moving to the next

### `code-reviewer.md`

The core review prompt used by both plan-review and session-review. Evaluation criteria:
- Correctness, security (OWASP top-10), DRY, test coverage
- Performance, code quality, navigability
- Specific, actionable, line-level feedback

### `plan-review.md`

Instructions for the spawned agent to dispatch an independent plan review subagent before implementing. Sends the plan + design doc + manifest to a review subagent with `code-reviewer.md` criteria.

### `session-review.md`

Same pattern as plan-review but for post-implementation code review. Sends the diff + plan to a review subagent.

### `receiving-code-review.md`

Instructions for how the spawned agent processes review feedback:
- Critically evaluate each suggestion — don't blindly agree
- Push back on incorrect suggestions with technical reasoning
- Accept valid improvements
- Verify technically, not performatively

## Orchestrator Package

The modular TypeScript application invoked by `/liteboard:run`.

### Module Breakdown

| Module | Responsibility |
|--------|---------------|
| `cli.ts` | Entry point, argument parsing |
| `types.ts` | Shared interfaces (Task, ModelConfig, Provider, etc.) |
| `parser.ts` | Manifest markdown -> Task[] |
| `resolver.ts` | Topological sort, dependency resolution, file conflict detection (see rules below) |
| `worktree.ts` | Git worktree create/cleanup, branch management |
| `spawner.ts` | Agent process spawning, stream parsing, log capture, stall detection |
| `merger.ts` | Trial merge, conflict resolution, build validation, merge serialization |
| `memory.ts` | memory.md append, snapshot building |
| `dashboard.ts` | Terminal UI rendering (progress bar, task status, per-task details) |
| `provider.ts` | Provider abstraction + Claude Code provider implementation |
| `brief.ts` | Assembles brief from commands/ + memory snapshot + task metadata |
| `progress.ts` | progress.md read/write, resume detection, failure summary extraction |

### CLI Interface

```bash
liteboard run <project-path-or-slug> [options]

Options:
  --concurrency=<N>       Max parallel agents, 1-5 (default: 1)
  --model=<model>         Override implementation model
  --branch=<name>         Feature branch name (default: liteboard/<slug>)
  --tasks=<1,2,3>         Run specific task IDs only
  --dry-run               Parse and show dependency graph only
  --verbose               Log all git commands to stderr
```

### Data Models

```ts
type TaskStatus =
  | "blocked"      // has unmet dependencies
  | "queued"       // dependencies met, awaiting worker
  | "running"      // agent process is live
  | "done"         // exited 0, squash merge complete
  | "failed";      // non-zero exit or merge conflict

interface Task {
  id: number;
  title: string;
  creates: string[];
  modifies: string[];
  dependsOn: number[];
  requirements: string[];
  tddPhase: string;
  commitMessage: string;
  complexity: number;          // 1-10
  status: TaskStatus;
  turnCount: number;
  lastLine: string;            // last agent output, max 120 chars
  bytesReceived: number;
  startedAt?: string;           // ISO 8601 timestamp
  completedAt?: string;         // ISO 8601 timestamp
  process?: ChildProcess;
  worktreePath?: string;
  logPath?: string;
}

interface ModelConfig {
  brainstorm:      { provider: string; model: string };
  taskManifest:    { provider: string; model: string };
  architectReview: { provider: string; model: string };
  implementation:  { provider: string; model: string };
  reviewGates:     { provider: string; model: string };
}

// Normalized stream events from any provider's output format.
// Each provider's parseStream() maps its native format to these.
type StreamEvent =
  | { type: "message_start"; turnIndex: number }
  | { type: "text_delta"; text: string }
  | { type: "tool_use_start"; toolName: string }
  | { type: "tool_use_end" }
  | { type: "message_end" }
  | { type: "error"; message: string };

interface Provider {
  name: string;
  spawn(opts: SpawnOpts): ChildProcess;
  parseStream(chunk: Buffer): StreamEvent[];
  healthCheck(): Promise<boolean>;
}

interface SpawnOpts {
  prompt: string;
  model: string;
  cwd: string;
  verbose: boolean;
}
```

## Provider Abstraction

V1 ships with the Claude Code provider only. The interface makes adding providers trivial.

### Claude Code Provider (V1)

Spawns:
```bash
claude -p <prompt> --dangerously-skip-permissions --output-format stream-json --verbose --model <model>
```

Parses `stream-json` output and maps to normalized `StreamEvent` types: `message_start`, `text_delta`, `tool_use_start`, `tool_use_end`, `message_end`, `error`.

### Stall Detection

Built into `spawner.ts`. Monitors bytes received from agent stdout:
- **Startup timeout:** 2 minutes with zero bytes received -> kill agent, mark failed. Likely cause: API throttling or CLI not responding.
- **Mid-task stall:** 5 minutes with no new bytes -> kill agent, mark failed. Likely cause: session limit hit or agent hung.

Stall detection runs on a 15-second polling interval per agent. On kill, the failure reason is written to `progress.md` for supervisor diagnosis.

### Future Providers (V2+)

- **Codex** — different CLI, different flags, different output format
- **Ollama via Claude Code** — `claude -p --model <ollama-model>` for locally-hosted models

### Per-Stage Model Config

Stored in `docs/liteboard/<slug>/config.json`:

```json
{
  "models": {
    "brainstorm":      { "provider": "claude", "model": "claude-opus-4-6" },
    "taskManifest":    { "provider": "claude", "model": "claude-opus-4-6" },
    "architectReview": { "provider": "claude", "model": "claude-opus-4-6" },
    "implementation":  { "provider": "claude", "model": "claude-opus-4-6" },
    "reviewGates":     { "provider": "claude", "model": "claude-sonnet-4-6" }
  },
  "concurrency": 1,
  "branch": "liteboard/<slug>"
}
```

Overridable via CLI flags. Changes mid-run affect future spawns only.

## Git Strategy

### Hard Rule: Never Touch Main

Liteboard never commits to or pushes to `main`. The feature branch is the only branch liteboard writes to. The user merges to `main` via PR.

### Branch Structure

```
main (never touched)
└── liteboard/<slug>                    <- feature branch
    ├── liteboard/<slug>-t1             <- task branch (ephemeral, in worktree)
    ├── liteboard/<slug>-t2
    └── liteboard/<slug>-t3
```

### Worktree Lifecycle

```
CREATE:  git worktree add -b <task-branch> /tmp/liteboard-<slug>-t1 liteboard/<slug>
         Always from latest feature branch HEAD.

WORK:    Agent commits freely in the worktree.

CLEANUP: git worktree remove --force <path>
         git branch -D <task-branch>
         Always runs — on success, failure, crash, everything.
```

### Merge Protocol

Every merge follows this protocol. The feature branch never contains an unvalidated commit.

```
Step 1: TRIAL MERGE
        git checkout liteboard/<slug>
        git merge --squash --no-commit <task-branch>

        Clean? -> Step 3.
        Conflict? -> Step 2.

Step 2: CONFLICT RESOLUTION
        a) package.json / lockfile?
           -> Accept both sides, run npm install, stage result.
        b) Other files?
           -> Abort merge.
           -> Squash task branch to single commit (git reset --soft, git commit).
           -> Rebase single commit onto feature branch (non-interactive).
           -> If rebase conflicts: abort immediately, mark task failed.
           -> If rebase succeeds: re-run trial merge.
           -> Still failing? Mark task failed, supervisor handles.

Step 3: VALIDATE
        Run npm run build in the merge result (pre-commit).
        Passes? -> Step 4.
        Fails? -> Abort merge, mark task failed with build error.

Step 4: COMMIT
        git commit -m "<exact task commit message>"

Step 5: CLEANUP
        Remove worktree + delete task branch.
```

### Merge Serialization

A proper mutex — one merge at a time. If a merge hangs for > 60s, kill it, abort, mark failed.

### Feature Branch Creation

On orchestrator startup:
- If `liteboard/<slug>` branch does not exist, create it from current `HEAD`
- If it already exists (from a previous run or resume), check it out and use it
- Never create from or reference `main` directly

### File Conflict Rules

Two tasks have a file conflict if any of their `creates` or `modifies` entries overlap. Specifically:
- Both `create` the same file -> conflict
- Both `modify` the same file -> conflict
- One `creates`, other `modifies` the same file -> conflict

Conflicting tasks must not run in parallel. The resolver treats file conflicts as implicit ordering constraints — the lower-ID task runs first.

### Crash Recovery (Startup Cleanup)

On orchestrator start:
1. List all git worktrees matching `/tmp/liteboard-*`
2. Remove any that don't correspond to a running task
3. Prune stale task branches
4. Read `progress.md` + git log to detect completed tasks
5. Resume from clean state

### Gitignore Management

On project creation, write `docs/liteboard/<slug>/logs/.gitignore` with `*`. Also add `.brief-t*.md` and `.memory-entry.md` to the repo's `.gitignore` so ephemeral files never pollute merges.

### Nuclear Option

If something truly goes wrong (corrupted merge state, detached HEAD):
```bash
git merge --abort
git checkout liteboard/<slug>
git reset --hard HEAD
```
Retry the task from scratch. The feature branch's committed history is always clean because we validate before committing.

## Supervisor Design

The supervisor is the behavior of the `/liteboard:run` skill after launching the orchestrator. It's what Claude does in the conversation while the build runs.

### Design Constraints

- Must survive hours in a single context window
- Every check-in must be context-cheap — read small files, tail logs only on failure
- Heavy quality work happens inside each task (plan-review, session-review subagents)

### Polling Mechanism

The `/liteboard:run` skill launches the orchestrator via the Bash tool with `run_in_background: true`. This returns immediately, freeing Claude to enter the supervisor loop.

Each check-in cycle:
1. Read `progress.md` via the Read tool (tiny file, ~50 lines)
2. If a task failed: tail the log via Bash (`tail -50 <log-path>`)
3. If intervention needed: use Bash to run git commands, restart tasks, etc.
4. Sleep for the adaptive interval via Bash (`sleep <N>`)
5. Repeat

This keeps each check-in to 2-3 tool calls maximum, preserving context window for hours of operation.

### Supervisor Loop

```
Launch orchestrator via Bash (run_in_background)
    |
    v
+-> Read progress.md via Read tool (~50 lines)
|   |
|   +-- Task completed?
|   |   -> Tail last 20 lines of log
|   |   -> Confirm: review subagents ran? tests passed? merge clean?
|   |   -> Report to user: "T3 done. 5/12 complete."
|   |
|   +-- Task failed?
|   |   -> Read failure summary from progress.md (cheap, first pass)
|   |   -> If more detail needed: tail last 50 lines of log
|   |   -> Diagnose: merge conflict? build error? agent crash? stall?
|   |   -> If retriable: restart task
|   |   -> If merge issue: fix merge, retry
|   |   -> If unrecoverable: flag to user with summary
|   |
|   +-- All tasks done?
|   |   -> Final summary, prompt user for PR
|   |   -> Exit supervisor
|   |
|   +-- Nothing new?
|       -> Brief status line: "8/12 done, T9+T11 running (14 min), no issues"
|
|   Adaptive wait:
|   - Task completed/failed in last check -> wait 5 min
|   - Steady state, tasks running normally -> wait 15 min
|   - Long-running tasks, no changes      -> wait 25 min
|
+---+
```

### What the Supervisor Does

- Reads `progress.md` (cheap) to track status
- Tails logs on failure for diagnosis (last 50 lines)
- Retries failed tasks (max 2 retries per task, 3 total attempts — then mark permanently failed and flag to user)
- Fixes merge issues
- Reports progress to user periodically
- Detects cross-task breakage (completed task broke something a later task depends on)

### What the Supervisor Does NOT Do

- Read full agent logs (too expensive for context)
- Evaluate code quality (that's the review subagents' job)
- Deeply analyze what agents are building
- Intervene in running tasks that aren't failing

## Shared Memory

### memory.md Structure

Append-only. The orchestrator appends memory entries after successful squash merge (not after agent exit). This prevents memory entries for work that failed to merge.

The agent writes `.memory-entry.md` in its worktree as its final step. The orchestrator reads this file from the worktree after squash merge succeeds (Step 4), then appends it to `memory.md` with a mutex to serialize concurrent writes.

```markdown
# Liteboard Memory — credential-vault

## T1 - Database Schema - 2026-03-10T14:52:00Z
**Created:** `src/db/schema.ts`, `tests/db/schema.test.ts`
**Exports:** `users`, `sessions`, `credentials` (Drizzle table definitions)
**Gotcha:** Uses Drizzle ORM, not raw SQL. All queries go through `src/db/index.ts`.

## T2 - Credential Vault - 2026-03-10T15:23:00Z
**Created:** `src/lib/credential-vault.ts`, `tests/lib/credential-vault.test.ts`
**Exports:** `encrypt(plaintext)`, `decrypt(encrypted, iv, authTag)`
**Gotcha:** Uses raw 256-bit key from ENCRYPTION_KEY env var. No PBKDF2.
```

Memory is injected into each agent's brief at spawn time (not at manifest generation time), so later tasks see entries from all previously completed tasks.

## Task Agent Workflow

Each spawned agent follows this mandatory workflow, enforced by `agent-orientation.md`:

### Phase 1: Explore & Plan
- Spawn Explore subagents (via Agent tool) to investigate the codebase in parallel
- Synthesize findings
- Write an implementation plan to `plan-t<id>.md`

### Phase 2: Plan Review
- Dispatch an independent plan review subagent (via Agent tool)
- Review subagent evaluates against `code-reviewer.md` criteria
- Agent processes feedback using `receiving-code-review.md` — pushes back on wrong suggestions, accepts valid ones
- Update plan with accepted changes

### Phase 3: Implement
- If TDD task: RED (failing tests) -> GREEN (minimum implementation) -> REFACTOR
- If non-TDD: implement per plan
- Run `npm run build` and `npm test` — fix failures

### Phase 4: Code Review
- Dispatch an independent code review subagent (via Agent tool)
- Same protocol as plan review — critical evaluation, no blind agreement
- Fix accepted issues, re-run build + tests

### Phase 5: Verify & Commit
- Run `npm run build` — verify no errors
- Run `npm test` — verify all tests pass
- Commit with exact task commit message
- Write `.memory-entry.md` with: files created, key exports, gotchas
- Delete plan file
- Do NOT push

## Testing

All orchestrator modules require tests (Vitest).

### parser.test.ts
- Parses all task fields from a sample manifest
- No-dependency tasks -> "queued", dependency tasks -> "blocked"
- Malformed manifest handled gracefully

### resolver.test.ts
- Topological sort produces correct layers
- Circular dependency detected and reported
- File conflict detection between tasks

### worktree.test.ts
- Creates worktree + task branch at correct path
- Cleanup on both success and failure
- Stale worktree cleanup on startup

### merger.test.ts
- Trial merge succeeds -> commit
- Trial merge conflicts on package.json -> auto-resolve
- Trial merge conflicts on other files -> rebase and retry
- Build validation before commit
- Merge serialization (mutex, one at a time)
- Timeout after 60s

### spawner.test.ts
- Turn count increments on message_start events
- lastLine strips markdown, truncates to 120 chars
- Exit 0 triggers merge + memory + dependency unlock
- Exit non-zero marks failed, does NOT unlock dependents
- Stall detection (no output timeout)

### memory.test.ts
- Concurrent appends are serialized (mutex, no corruption)
- Snapshot includes all tasks completed at snapshot time
- Entry format validates: timestamp, task id, created files, gotchas

### progress.test.ts
- Atomic write (temp + rename)
- Resume detection from progress.md
- Resume detection from git log
- Failure summary extraction

### brief.test.ts
- agent-orientation.md always prepended
- Memory snapshot injected at spawn time
- Brief components composed correctly
- Task metadata included

### dashboard.ts
Test-exempt. Pure terminal rendering — tested manually. No unit tests required.

## V1 Scope

**In scope:**
- All three user-invocable skills (brainstorm, task-manifest, run)
- All five brief components
- Full modular orchestrator with all modules listed above
- Claude Code provider only
- Bulletproof git merge protocol
- Baked-in supervisor with adaptive monitoring
- Per-stage model config via config.json
- Resume support
- npm package + Claude Code plugin distribution
- Tests for all orchestrator modules

**Out of scope (V2+):**

- Ollama/local model providers
- Loop detection / scope creep detection (automated)
- Ink-based TUI (terminal dashboard is sufficient)
- Web UI
- `liteboard list` / `liteboard resume` CLI commands
